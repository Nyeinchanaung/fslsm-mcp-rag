## 2. Database Design — PostgreSQL + pgvector

**Why PostgreSQL (not just JSON files)?** Your paper mentions ~2,500 chunks now and plans to add textbooks later. PostgreSQL with pgvector gives you scalable vector search, relational joins between chunks/profiles/logs, ACID transactions for simulation logs, and easy migration when the corpus grows. JSON files are fine for debugging snapshots but not for production experiments with 4,000 sessions.

### Schema Design

```sql
-- Enable vector extension
CREATE EXTENSION IF NOT EXISTS vector;

-- ============================================================
-- CORPUS & RETRIEVAL
-- ============================================================
CREATE TABLE textbooks (
    id          SERIAL PRIMARY KEY,
    name        VARCHAR(255) NOT NULL,       -- "Dive into Deep Learning"
    version     VARCHAR(50),                 -- "v2.0"
    source_path TEXT,                        -- path to raw file
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE chunks (
    id              SERIAL PRIMARY KEY,
    chunk_uid       VARCHAR(64) UNIQUE NOT NULL,  -- e.g., "309b6eee8af4_0"
    textbook_id     INT REFERENCES textbooks(id),
    chapter         VARCHAR(100),
    section         VARCHAR(200),
    concept_labels  TEXT[],                  -- array of concept tags
    content         TEXT NOT NULL,           -- raw chunk text
    embedding       vector(384),             -- dim depends on model (384 for MiniLM)
    token_count     INT,
    created_at      TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_chunks_embedding ON chunks
    USING ivfflat (embedding vector_cosine_ops) WITH (lists = 50);

CREATE INDEX idx_chunks_textbook ON chunks(textbook_id);

-- ============================================================
-- FSLSM PROFILES
-- ============================================================
CREATE TABLE fslsm_profiles (
    id              SERIAL PRIMARY KEY,
    profile_code    VARCHAR(20) UNIQUE NOT NULL,  -- e.g., "ASIVS" (Act,Sen,Intuit,Vis,Seq)
    act_ref         SMALLINT NOT NULL CHECK (act_ref IN (-1, 1)),
    sen_int         SMALLINT NOT NULL CHECK (sen_int IN (-1, 1)),
    vis_ver         SMALLINT NOT NULL CHECK (vis_ver IN (-1, 1)),
    seq_glo         SMALLINT NOT NULL CHECK (seq_glo IN (-1, 1)),
    description     TEXT,                    -- human-readable label
    style_descriptor TEXT                    -- Graf (2007) style description
);
-- Pre-populate with 16 rows (2^4 combinations)

-- ============================================================
-- AGENTS (EXPERIMENT 1+)
-- ============================================================
CREATE TABLE agents (
    id              SERIAL PRIMARY KEY,
    agent_uid       VARCHAR(20) UNIQUE NOT NULL,  -- "agent_P01_I03"
    profile_id      INT REFERENCES fslsm_profiles(id),
    instance_num    SMALLINT NOT NULL,       -- 1..5
    llm_model       VARCHAR(100),            -- "gpt-4o", "claude-sonnet", "llama3-8b"
    system_prompt   TEXT,                    -- full generated prompt
    created_at      TIMESTAMPTZ DEFAULT NOW()
);

-- ============================================================
-- GROUND TRUTH (MULTI-HOP QA)
-- ============================================================
CREATE TABLE eval_questions (
    id              SERIAL PRIMARY KEY,
    question_id     VARCHAR(30) UNIQUE NOT NULL,  -- "d2l_mh_0001"
    question        TEXT NOT NULL,
    gold_answer     TEXT NOT NULL,
    answer_type     VARCHAR(50),             -- "synthesize_workflow", "compare", etc.
    difficulty      VARCHAR(20),
    quality_tier    VARCHAR(10),             -- "gold" / "silver"
    gold_chunk_ids  TEXT[] NOT NULL,          -- array of chunk_uids
    essential_ids   TEXT[],
    supporting_ids  TEXT[]
);

-- ============================================================
-- EXPERIMENT LOGS
-- ============================================================
CREATE TABLE experiment_runs (
    id              SERIAL PRIMARY KEY,
    experiment      VARCHAR(10) NOT NULL,    -- "exp1", "exp2", "exp3"
    config          JSONB,                   -- full config snapshot
    started_at      TIMESTAMPTZ DEFAULT NOW(),
    finished_at     TIMESTAMPTZ
);

CREATE TABLE interaction_logs (
    id              SERIAL PRIMARY KEY,
    run_id          INT REFERENCES experiment_runs(id),
    agent_id        INT REFERENCES agents(id),
    question_id     INT REFERENCES eval_questions(id),
    system_variant  VARCHAR(10),             -- "R0"/"R1" (exp2), "S0"/"S1a"/"S1b" (exp3)
    
    -- Request/Response
    prompt_tokens   INT,
    completion_tokens INT,
    latency_ms      FLOAT,
    
    -- Retrieved context
    retrieved_chunk_ids TEXT[],
    selected_tool   VARCHAR(100),            -- for exp3
    
    -- Raw outputs
    tutor_response  TEXT,
    agent_response  TEXT,                    -- student engagement feedback
    
    -- Computed metrics (nullable — filled by evaluation)
    pra_score       FLOAT,                   -- exp1
    das_scores      JSONB,                   -- exp1: per-dimension
    scs_score       FLOAT,                   -- exp2
    rr_score        FLOAT,                   -- exp2
    engagement      FLOAT,                   -- exp2
    chunk_recall_5  FLOAT,                   -- exp2
    tsa_correct     BOOLEAN,                 -- exp3
    
    created_at      TIMESTAMPTZ DEFAULT NOW()
);
```

### Why Not Just FAISS?

FAISS is still used for the **in-memory vector index** during RAG retrieval at runtime. PostgreSQL with pgvector serves as the **persistent storage + metadata layer**. The workflow is: chunks live in PostgreSQL → at experiment start, load embeddings into FAISS for fast retrieval → log results back to PostgreSQL. This gives you the best of both worlds.

## 4. Scalable Corpus Ingestion Pipeline

The ingestion pipeline is designed so adding a new textbook = one command.

### Pipeline Steps

```
Raw PDF/HTML  →  Parser  →  Chunker  →  Embedder  →  Database
                  │           │            │            │
              plain text   ~500 token    vector       chunks table
                          chunks w/     (384-dim)    + pgvector index
                          metadata
```

### Adding a New Textbook Later

```bash
# 1. Drop the new textbook into data/raw/
cp new_textbook.pdf data/raw/new_textbook/

# 2. Run the ingestion pipeline with a textbook name
python scripts/ingest_d2l.py \
    --source data/raw/new_textbook/ \
    --textbook-name "Machine Learning: A Probabilistic Perspective" \
    --chunk-size 500 \
    --chunk-overlap 50

# 3. The pipeline:
#    - Creates a new row in `textbooks`
#    - Parses, chunks, embeds
#    - Inserts into `chunks` with textbook_id foreign key
#    - Rebuilds FAISS index from all chunks
```

No code changes needed — the `textbook_id` foreign key keeps corpora separate while allowing cross-textbook retrieval when desired.

## 5. Experiment 1 — Detailed Implementation Plan

### Objective Recap
Validate that 80 virtual LLM agents (16 profiles × 5 instances) consistently reflect their assigned FSLSM profiles by answering the 44-item ILS questionnaire.

### Implementation Steps

**Step 1: Profile & Agent Setup**
- Load 16 profiles from `profiles.json` → seed into `fslsm_profiles` table
- `agent_factory.py` generates 80 agent records: for each profile, create 5 instances with the system prompt template from Appendix A.2.1
- Store agent records in `agents` table with full system prompts

**Step 2: ILS Questionnaire Runner**
- `ils_evaluator.py` iterates through all 80 agents
- For each agent: sends the 44 ILS questions using the prompt from Appendix A.2.2
- Collects (a)/(b) responses, stores raw responses in `results/exp1/raw_responses/`
- Repeat for N independent trials (paper suggests multiple stability checks)

**Step 3: Scoring**
- Standard ILS scoring: each question maps to one FSLSM dimension
- Options (a) and (b) map to opposite poles (-1 or +1)
- Sum per dimension → score on -11 to +11 scale
- "Match" = if scored pole matches assigned binary profile

**Step 4: Metrics Computation**
- **Profile Recovery Accuracy (PRA)**: proportion of dimensions where detected profile matches assigned profile, averaged across agents
- **Dimension Alignment Score (DAS)**: per-dimension agreement rate across all agents and trials
- Store in `interaction_logs` and export to CSV

**Step 5: Visualization**
- Radar charts: predefined index vs. questionnaire result per agent (as in Figure 3.4)
- Heatmap: 16 profiles × 4 dimensions showing alignment rates
- Model comparison: run across GPT-4o, Claude Sonnet, Llama 3 8B

### Experiment 1 Config (`experiments/exp1_agent_fidelity/config.yaml`)

```yaml
experiment: exp1_agent_fidelity
description: "Fidelity of Virtual Student Agent Encoding (RQ2)"

agents:
  num_profiles: 16
  instances_per_profile: 5
  total_agents: 80

  - name: gpt-4o
    provider: openai
    temperature: 0.3
  - name: claude-sonnet
    provider: anthropic
    temperature: 0.3
  - name: llama3-8b
    provider: huggingface
    temperature: 0.3

  num_questions: 44
  num_trials: 3          # repeat for stability check

  metrics:
    - profile_recovery_accuracy
    - dimension_alignment_score
  visualization:
    - radar_chart_per_agent
    - heatmap_profiles_x_dimensions
    - model_comparison_bar_chart
```

## 8. Key Design Decisions Summary

| Decision | Recommendation | Rationale |
|----------|---------------|-----------|
| **Chunk storage** | PostgreSQL + pgvector (not just JSON) | Scalable, queryable, supports vector search, relational joins for logs |
| **FSLSM profiles** | JSON seed file → PostgreSQL | Version-controlled source of truth + runtime queryability |
| **Vector search** | pgvector for persistence, FAISS for runtime | Best of both: durable storage + fast in-memory retrieval |
| **LLM client** | Unified abstraction over OpenAI/Anthropic/HF | Experiment 1 compares 3 models — need seamless switching |
| **Logging** | Structured JSONL + PostgreSQL `interaction_logs` | JSONL for streaming/debugging, DB for aggregation queries |
| **Experiment isolation** | Separate `experiments/expN/` folders with own config | Clean separation, reproducible runs, independent configs |
| **Scalability** | `textbooks` FK on `chunks`, pipeline per textbook | Adding a corpus = one CLI command, no code changes |
